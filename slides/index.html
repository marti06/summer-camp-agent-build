first principles

What even is an agent?

An LLM calling tools in a loop until the task is done.
ask the model (with tools) → wants a tool? run it, feed it back → repeat → done

That's the whole concept. Everything else is how much of that loop you write yourself.

Why agents exist, what they buy you

  • The task is open-ended, no fixed script you could hardcode.
  • The model can use tools & react, search, read, write, call APIs.
  • You can verify the result, so iteration is safe.
  • They do real work while you don't watch, that's the leverage.

…and when NOT to

  • If a single prompt does it, use a single prompt.
  • If the steps are fixed, use a workflow, not an agent.
  • Reach for an agent only when the task is open-ended enough to earn it.

(This is also why an agent that does arithmetic is silly. A calculator is a function call, not an agent.)

how we got here, fast

Two years,
four leaps

From a loop to a platform

~2023One LLM call. Prompt in, text out. No tools, no memory.
thenThe hand-rolled loop. Give it tools, write the while-loop yourself. The first "agent." Powerful, and fiddly.
thenWorkflows. Orchestrate calls on fixed paths, chain, route, parallelize. Predictable. The raw loop stopped being something you wrote by hand.
thenDynamic / agentic. The model drives its own path & tools. Workflows that decide their own steps.
nowThe platform. Claude Code, subagents, MCP, the Agent SDK. The loop, the workflows, the orchestration, given to you. You compose; you don't re-implement.

The actual skill: evolve alongside it

  • Each leap retired hand-written plumbing from the leap before.
  • The winners stop re-implementing what the platform now gives them…
  • …and spend that attention on tools, composition, and shipping.
Today we ride the abstraction all the way up, and build at the top of it.

Three ideas we'll prove with code

  • An agent is committable config, a markdown file, not a script.
  • An orchestration is composition, agents fanning out + a judge.
  • Deploy = git commit, and your product's MCP can be the agent's tools.

Setup, one key

  • Claude Code installed · git · the repo cloned
  • ANTHROPIC_API_KEY, provided here. Web search & fetch are built in; no extra key.
  • Stacklist credential, optional, only to publish for real (else it writes a local file)
cp .env.example .env      // fill in ANTHROPIC_API_KEY
set -a && source .env && set +a
claude                    // open Claude Code in the repo

phase 1

An agent is a file

---
name: link-researcher
description: Finds strong candidate links for a topic. Use this first.
tools: WebSearch, WebFetch     // blast radius: search & fetch, nothing else
model: sonnet
---
You are a research specialist. Given a TOPIC, find the strongest...

description = how the lead delegates. tools = what it's allowed to touch.

phase 2

A panel beats one opinion

  • Three reviewers: credibility · freshness · fit
  • Each owns ONE lens · same cheap model · run in parallel
  • Independent contexts → real disagreement to resolve

phase 3

Orchestrate, then judge

topic → researcher → credibilityfreshnessfit → judge → stack

/build-stack fans out in parallel; the lead agent is the judge.

phase 4 · the product moment

Your product = the agent's tools

  • Default: writes output/<slug>.md, works with no account
  • Connect the Stacklist MCP + allow its tools → it publishes for real
  • + an AI-discoverability check on the way out

The agent didn't change, you widened its allowlist to include your product.

phase 5 · the payoff

Deploy = git commit

git add .claude CLAUDE.md && git commit -m "Add /build-stack" && git push
// teammate:
git pull && claude
> /build-stack "self-hosting your analytics"     // zero setup

Same agent, headless in CI (claude -p) on a schedule. Wrote it once.

What bites in production

  • Tool sprawl, if a human can't pick the tool, neither can the model
  • Context rot, histories fill, recall degrades
  • Vague descriptions, that's the delegation prompt
  • Cost, orchestration multiplies tokens; tier your models

Keep climbing

  • Give panelists real tools · loop-to-consensus · a code-review panel on your repo
  • Embed it: the Agent SDK · agent teams for debate
  • The platform will move again, build at the top rung, every time
